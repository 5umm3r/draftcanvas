# 画像生成モデル調査 — 2026-05-09

## 1. トップバー pill（5h/weekly）の表示内容

5h セッション（時計アイコン）と weekly セッション（カレンダーアイコン）は **ChatGPT/Codex アカウントのレートリミット残量**を表示。

- 表示値: `100 - usedPercent` の残量パーセント（四捨五入）
- リセット時刻: `あと 3h12m` / `あと 2d 5h` 形式
- データ取得: JSON-RPC `account/rateLimits/read` → `rateLimitsByLimitId.codex`（フォールバック: `rateLimits`）

関連コード:
- `Models.swift:578-720` — `CodexAccountUsageStatus` 構造体・パースロジック
- `ContentView.swift:185-196` — `usagePill(...)` ×2 の表示
- `CodexAppServerClient.swift:222-237` — `readAccountUsageStatus()` 呼び出し

## 2. Codex / ChatGPT レートリミットの関係

**別枠管理**。片方を使い切っても他方には影響しない。

- API レスポンスが `rateLimitsByLimitId: { "codex": {...}, "chatgpt": {...} }` の dict 構造 → 複数バケット前提の設計
- 本アプリの pill は **Codex バケットのみ**反映
- ブラウザ ChatGPT のチャット枠（chatgpt バケット）は pill に表示されない

| 状況 | このアプリ（Codex） | ブラウザChatGPT |
|------|------------------|----------------|
| Codex 5h 100%消費 | ブロック | 使える |
| ChatGPT 5h 100%消費 | 使える | ブロック |

## 3. 画像生成のフロー

```
[アプリ]
  → JSON-RPC turn/start
[Codex CLI app-server]
  → OpenAI Responses API
    → image_generation_call ツール
      → [画像モデル（内部選択）]
        → Base64 PNG を返す
```

- OpenAI Images API (`/v1/images`) 直接呼び出し経路は**存在しない**
- 画像生成はプロンプト指示で Codex にツール呼び出しを促す方式
- 画像モデル名はアプリから一切指定しない（`GenerationCoordinator.swift:114-160`）

関連コード:
- `GenerationCoordinator.swift:74` — `client.startThread(model: request.model, ...)`
- `CodexAppServerClient.swift:209-215` — `turn/start` JSON-RPC 送信
- `JSONRPCCodec.swift:64-82` — `extractImageResult`（`image_generation_call` レスポンスからBase64取得）

## 4. アプリ内「モデル選択」の意味

アプリで選択するモデル = **Codex 親エージェントモデル**（gpt-5/o系の推論モデル）。

`gpt-image-*`（画像モデル）**ではない**。

| 変わるもの | 変わらないもの |
|-----------|--------------|
| revised_prompt の整形品質 | 最終画像のスタイル/品質 |
| 推論強度（reasoningEffort） | 画像モデル自体 |
| ツール呼び出しの判断 | |
| トークン消費・レート枠消費量 | |

設計: `ContentView.swift:780-808` (Picker) → `CodexAppServerClient.swift:140-153` (`thread/start` の `model` params)

## 5. 画像モデル検証結果（2026-05-09 実測）

### 検証手順

```bash
codex exec --skip-git-repo-check "とても小さい赤い円の画像を生成して"
```

生成されたPNG（`~/.codex/sessions/2026/05/09/rollout-*.jsonl`）の C2PA メタデータを抽出:

```
softwareAgent:
  name:    gpt-image
  version: pre-2.0
claim_generator_info:
  name: OpenAI Media Service API
```

### 結果サマリ

| 項目 | 値 |
|------|---|
| Codex CLI バージョン | 0.128.0 |
| 親モデル | gpt-5.5（`~/.codex/config.toml`） |
| 画像モデル | **gpt-image pre-2.0** |
| プロバイダ | OpenAI Media Service API |
| 認証 | Codex OAuth（ChatGPTサブスク、API key不要） |
| session ID | 019e0ce2-0383-7360-a7f9-4e4058800839 |

### `pre-2.0` の解釈

- **有力**: gpt-image-2 プレリリース版（gpt-image-2 は 2026-04-21 リリース、3週間後に検証）
- 代替: gpt-image-1.5 が内部的に pre-2.0 タグを持つ（断定不可）
- 確実なのは gpt-image-1 ではない

## 6. gpt-image-2 の Codex 対応状況

- OpenAI 公式: 2026-04-21 リリース。タイトル「**Introducing gpt-image-2 — available today in the API and Codex**」
- Codex OAuth 経由で画像生成が動作することを確認（API key 不要）
- openclaw/openclaw issue（OAuth不可という報告）はあのフォーク固有の問題

## 7. 画像モデル切替機能の実装可能性

| 案 | 概要 | 難度 | 備考 |
|----|------|------|------|
| A | Codex `config` 経由で画像モデル指定 | **困難** | Codex CLI 側プロトコル対応未確認 |
| B | OpenAI Images API 直叩き（Codexバイパス） | **中** | API key管理・課金体系切替・HTTP client新規実装が必要 |
| C | テキスト=Codex / 画像=直叩き 併用 | **中** | 案Bにコンテキスト分離コスト追加 |

**結論**: 現状すでに最新モデル（pre-2.0）が使用されているため実装緊急性は低い。  
透過背景が必要な場合のみ `gpt-image-1.5 --background transparent` への切替パスに価値あり（`gpt-image-2` は `background=transparent` 非対応）。

## 参考: C2PA メタデータ抽出方法

```python
import json, base64, re

with open("rollout-*.jsonl") as f:
    for line in f:
        d = json.loads(line)
        if d.get("type") == "response_item" and d["payload"].get("type") == "image_generation_call":
            result = d["payload"]["result"]
            data = base64.b64decode(result)
            for kw in [b"gpt-image", b"softwareAgent", b"OpenAI"]:
                idx = data.find(kw)
                if idx >= 0:
                    ctx = data[max(0, idx-50):idx+150]
                    print(re.sub(rb"[^ -~]", b".", ctx).decode("ascii", errors="replace"))
```
